## 最终输出格式

所有提取的字段必须整合为一个JSON对象，输出为纯JSON格式（不包含Markdown标记）。

### 完整输出结构

```json
{
  "basic_info": "字符串",
  "highlights": "字符串",
  "itinerary_days": [...],
  "cost_included": [...],
  "cost_excluded": [...],
  "notices": [...],
  "index_text": "字符串",
  "index_tags": [...],
  "certificate_limit": "字符串",
  "age_limit": "字符串"
}
```

### 字段说明

| 字段名 | 类型 | 必需 | 缺失处理 |
|--------|------|------|----------|
| basic_info | 字符串 | 是 | "未知" |
| highlights | 字符串 | 是 | "未提及亮点" |
| itinerary_days | 数组 | 是 | [] |
| cost_included | 数组 | 是 | [] |
| cost_excluded | 数组 | 是 | [] |
| notices | 数组 | 是 | [] |
| index_text | 字符串 | 是 | "" |
| index_tags | 数组 | 是 | [] |
| certificate_limit | 字符串 | 是 | "未提及证件限制" |
| age_limit | 字符串 | 是 | "未提及年龄限制" |

### 强制规则

1. **纯JSON输出**：
   - 不使用Markdown代码块标记（```json ... ```）
   - 不添加任何解释性文字或说明
   - JSON必须可直接解析，无语法错误

2. **字段完整性**：
   - 所有字段必须存在于输出中
   - 缺失信息按照上表默认值填充
   - 不得遗漏任何字段

3. **格式一致性**：
   - 严格按照各模块的提取规范
   - JSON结构固定，不可增删字段
   - 所有引号、逗号、括号必须正确配对

## 基础信息与亮点提取

### 提取目标
从旅游行程文档中提取两个独立字符串变量：`basic_info` 和 `highlights`

### basic_info 字段规范

**包含信息（按顺序整合）**：
1. **行程主标题**：
   - 优先使用文档中的原标题
   - 若无明确标题，自动生成格式：`目的地（国家-城市/地区）名称+总天数+总晚数`
   - 若目的地未知：使用`未知+总天数+总晚数`

2. **目的地信息**：
   - 包含国家和城市全名（可多个）
   - 若仅提及城市，国家标记为"未知"
   - 格式：`国家-城市`

3. **行程时长**：
   - 总天数：数字+`天`
   - 总晚数：数字+`晚`
   - 若未标注，参考每日行程推算

**输出格式**：
```
basic_info: 行程标题为"[标题/目的地+天数+晚数]"，目的地为"[国家-城市]"，共[天数]天[晚数]晚。
```

**示例**：
```
basic_info: 行程标题为"日本大阪-东京5天4晚"，目的地为"日本-大阪、东京"，共5天4晚。
```

### highlights 字段规范

**提取规则**：
1. **优先提取明确标注内容**：
   - 关键词：`亮点`、`特色`、`特点`、`★`
   - 按原文顺序排列，不改写、不概括

2. **若无明确标注**：
   - 分析整体行程内容
   - 总结3-5个代表性亮点
   - 使用原文表述概括

**输出格式**：
```
highlights: 亮点包含"亮点1原文内容"、"亮点2原文内容"
```

**示例**：
```
highlights: 亮点包含"全程入住五星级酒店"、"专业中文导游陪同"、"包含所有景点门票"
```

### 强制规则
- 仅使用文档原文，禁止推测或补充
- 每日行程、费用包含、注意事项等不在highlights中输出
- 缺失信息用"未知"填充

## 每日行程信息提取

### 输出格式（JSON）

```json
{
  "itinerary_days": [
    {
      "day": 1,
      "day_title": "标题内容",
      "meals": {
        "breakfast": "早餐安排",
        "lunch": "午餐安排",
        "dinner": "晚餐安排"
      },
      "hotel": {
        "hotel_name": "酒店名称",
        "hotel_level": "酒店级别"
      },
      "pois": [
        {
          "poi_name": "POI名称",
          "poi_type": "POI类型",
          "activity": "活动描述"
        }
      ]
    }
  ]
}
```

### 字段说明

#### day（必需）
- 类型：数字
- 提取规则：
  - 优先提取原文中的天数标识
  - 若无明确标注，按行程顺序自动编号（1、2、3...）

#### day_title（必需）
- 类型：字符串
- 提取规则：
  - 提取当日行程的标题（城市串联、主题描述）
  - 若未明确标注，填`"未提及"`
  - 第一天通常为前往目的地，可直接解析为目的地

#### meals（必需）
- 类型：对象
- 子字段：
  - `breakfast`：早餐安排
  - `lunch`：午餐安排
  - `dinner`：晚餐安排
- 缺失规则：未提及的字段填`"未提及"`

#### hotel（必需）
- 类型：对象
- 子字段：
  - `hotel_name`：酒店名称
  - `hotel_level`：酒店级别
- 缺失规则：未提及的字段填`"未提及"`

#### pois（必需）
- 类型：数组
- POI定义：景点、城市、活动、体验等
- 字段说明：
  - `poi_name`：POI名称（如"故宫博物院"）
  - `poi_type`：类型（如"景点"、"城市"、"文化体验"）
  - `activity`：活动描述（1句话，如"参观故宫博物院"）

### 提取规则
1. 按天拆分，不合并、不遗漏
2. POI中的多个同类活动需拆分为独立对象
3. 所有字符串字段仅在内容明确时填写
4. 缺失信息严格填`"未提及"`

### 特殊处理
- 日期不连续：按逻辑拆分，重复内容以最后出现的记录为准
- 格式错误：异常部分标注为`"未提及"`

## 费用条款与注意事项提取

### 输出格式（JSON）

```json
{
  "cost_included": [
    "费用包含项1",
    "费用包含项2"
  ],
  "cost_excluded": [
    "费用不包含项1",
    "费用不包含项2"
  ],
  "notices": [
    "注意事项1",
    "注意事项2"
  ]
}
```

### cost_included 提取规范

**提取范围**：
- 文档中直接标注"费用包含"的费用项
- 明确列出的费用项目（交通、门票、住宿等）

**排除内容**：
- 行程安排描述
- 酒店详情
- 每日活动内容

**提取规则**：
- 必须提取原文条目，不合并、不改写
- 若无"费用包含"标注，仅提取明确提及的费用项
- 无费用项时数组留空

### cost_excluded 提取规范

**提取范围**：
- 文档中直接标注"费用不包含"的非费用项
- 明确列出的非费用项目（个人消费、自费项目、景点内购物等）

**排除内容**：
- 行程安排描述
- 酒店详情
- 每日活动内容

**提取规则**：
- 必须提取原文条目，不合并、不改写
- 若无"费用不包含"标注，仅提取明确提及的非费用项
- 无非费用项时数组留空

### notices 提取规范

**提取范围**：
- 重要提示（建议携带证件）
- 免责声明（天气取消不承担责任）
- 行为限制（禁止私自离团）

**排除内容**：
- 行程描述
- 酒店信息
- 费用项

**提取规则**：
- 必须提取原文表述，不合并、不改写
- 仅保留与"注意事项"直接相关的内容
- 无注意事项时数组留空

## 索引摘要生成

### 输出格式

```
index_text：[60~120字结构化文本]
index_tags：[JSON数组格式的标签列表]
certificate_limit：[证件限制说明]
age_limit：[年龄限制说明]
```

### index_text 规范（60~120字）

**必须包含的核心要素**：
1. 目的地（国家/城市/区域）
2. 行程时长（几天几晚，无明确天数写"天数未知"）
3. 2~4个核心体验/主题玩法
4. 2~3个关键行程点
5. 2~3个费用包含关键词

**禁用内容**：
- 营销话术（"超值"、"必去"）
- 主观评价（"推荐"、"值得"）
- 未提及的补充信息

**语言风格**：
- 口语化、简洁
- 避免长句和专业术语
- 信息密度≥80%

**示例**：
```
index_text：厦门5天4晚，适合亲子家庭，含鼓浪屿打卡、环岛路骑行、曾厝垵美食，行程含门票+住宿，主题玩法：海岛+文化体验
```

### index_tags 规范（10~25个标签）

**覆盖维度（必填）**：
- 目的地标签
- 时长标签
- 人群标签
- 主题标签
- 核心场景标签
- 费用标签

**生成规则**：
- 每个标签≤4字
- 允许合理同义词扩展（需原文语义支持）
- 无明确信息时用"未知"占位

**示例**：
```json
["厦门", "5天4晚", "亲子", "鼓浪屿", "环岛路", "曾厝垵", "海岛", "文化", "美食", "骑行", "门票", "住宿", "家庭游", "短途游"]
```

### certificate_limit 规范

**提取范围**：
- 护照要求
- 签证要求
- 港澳通行证
- 身份证
- 健康证明

**输出规则**：
- 完全来自原文语义
- 未提及时输出：`未提及证件限制`

**示例**：
```
certificate_limit：需持有效身份证办理入住，出境游需护照
```

### age_limit 规范

**提取范围**：
- 最低年龄
- 最高年龄
- 儿童政策
- 老人限制
- 婴幼儿限制

**输出规则**：
- 完全来自原文语义
- 未提及时输出：`未提及年龄限制`

**示例**：
```
age_limit：2岁以上儿童可报名，70岁以上需家属陪同
```

### 字符容错原则

**常见问题处理**：
1. **乱码符号**：忽略"??""�"等乱码
2. **换行错位**：根据上下文修复断句
3. **表格结构丢失**：通过语义理解重建
4. **标点缺失**：根据内容逻辑补充
5. **标题层级混乱**：按语义识别重要性

**处理规则**：
- 不因格式错乱丢弃有效信息
- 可推断的信息通过上下文语义修复
- 严禁编造未出现的信息
- 仅在语义无法判断时使用"未知"

## 通用强制规则

1. **信息真实性**：
   - 所有内容必须100%来自输入文本语义
   - 严禁编造、补充未提及的信息

2. **格式一致性**：
   - 严格按照指定格式输出
   - JSON格式需可直接解析
   - 无额外说明文字

3. **容错能力**：
   - 具备处理乱码、格式错位的能力
   - 通过语义理解恢复结构
   - 不因格式问题丢弃有效信息
